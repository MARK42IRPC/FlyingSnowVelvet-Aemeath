飞行雪绒音乐模块多源优化方案计划
================================

一、目标
1. 统一 QQ / 酷狗 / 网易云 provider 边界，降低宿主层对平台细节的感知。
2. 引入多源搜索路由与失败回退，提升搜索可用性与命中率。
3. 记录 provider 健康度，为后续播放 URL 回退、歌单同步、插件化做准备。

二、分阶段改造计划
阶段 1：统一协议层
- 梳理现有 `lib/script/music` 抽象层，补齐 provider 标签、能力声明、模式支持判断。
- 将宿主统一入口固定为 `MusicService`，避免 UI / 工具层继续直接感知平台实现。

阶段 2：搜索路由层
- 新增 `SourceRouter`，默认优先当前平台，再按配置顺序回退其它平台。
- 搜索结果使用统一去重规则，避免相同歌曲在多源下重复展示。
- 在混合结果中标注来源平台，减少用户误判。

阶段 3：健康度治理
- 记录 provider 搜索耗时、成功 / 失败次数、连续失败次数、最近错误信息。
- 回退顺序优先健康度更好的 provider，弱化失效平台影响。

阶段 4：播放回退（后续）
- 当当前平台 URL 解析失败时，基于标题 / 歌手 / 时长尝试跨源匹配备用音源。
- 在播放器侧增加“保留原平台优先，跨源仅兜底”的策略开关。

阶段 5：登录与资料库治理（后续）
- 统一二维码登录状态机、Cookie 持久化、登录失效探测。
- 抽象喜欢列表 / 歌单同步模型，减少 provider 专属分支散落。

阶段 6：外置 provider（后续）
- 将 QQ / 酷狗适配器拆分为独立模块或独立仓库，向插件化靠拢。

三、本次改造落地范围
1. 完成阶段 1：补齐 provider 协议元信息。
2. 完成阶段 2：新增多源搜索路由与去重。
3. 完成阶段 3：新增 provider 搜索健康度统计。
4. UI / 工具搜索结果增加来源标识。

四、验收标准
1. 当前平台搜索正常返回时，结果优先来自当前平台。
2. 当前平台结果不足或失败时，可自动补充其它平台结果。
3. 混合结果不会大量重复歌曲。
4. 搜索展示可识别来源平台。
5. 现有播放、登录、历史记录功能不被破坏。
